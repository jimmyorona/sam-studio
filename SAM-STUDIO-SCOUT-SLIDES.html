3 / 14 Architecture

System Architecture

Express + Vue SPA with Python subprocess helpers. Local Ollama model backend. No external dependencies required.

sam-studio/ ├── personas/ → 18 canonical persona files (.md) ├── scripts/ │ ├── pptx_to_video.py → narration + TTS + video assembly │ ├── reviewer_synth.py → review/rewrite fan-out + export │ └── extract.py → PPTX/PDF/DOCX/MD/TXT → markdown ├── web/ │ ├── server/index.js → Express :3001 (all modes, SSE progress) │ └── src/ → Vue 3 SPA (4-tab UI) └── reviews/ → generated reports (gitignored)

🔒 Local-First

All generation against local Ollama model. No document content traverses external boundaries unless explicitly configured.

🔊 TTS Options

Edge TTS (free), ElevenLabs (premium), or offline Supertonic. Per-persona voice recommendations included.

7 / 14 Prompt Actions

REVIEW-DOCUMENT.md

Constrained output templating with persona injection.

Design Pattern

Interpretation Hierarchy

AUTHORITATIVE: ## Review Lens section
ADOPT: Character Profile, tone, guardrails
IGNORE: TTS mechanics (ElevenLabs, SSML)

Fixed Output Schema

Verdict → Strengths → Findings (with severity) → Questions → Scorecard — same bones regardless of persona.

Architectural Strengths

  • Self-referential quality bar: Each persona's scorecard defines what "good" means for that lens
  • Comparable outputs across personas: Fixed structure enables synthesis of multiple reviews
  • Graceful degradation: Missing Review Lens → infer from archetype; never fails
  • Severity calibration per persona: Different experts define "Critical" differently — by design
8 / 14 Prompt Actions

REWRITE-DOCUMENT.md

Voice transformation with structural license and fact preservation.

Design Pattern

Quality Constraint

"The rewrite must score 5/5 on the persona's own scorecard." Self-referential closed-loop quality definition.

Fact Preservation

NEVER invent data. Missing info → [NEEDS: ...] placeholder. Escape valve without fabrication.

Key Capabilities

  • Structural freedom: May reorder, merge, split slides — with provenance tracking
  • Format-aware: Presentations get speaker notes; prose stays prose
  • Persona-driven placeholders: Some personas draft proposals; some insert strict markers
  • Rewrite notes section: Documents all structural changes and lists all [NEEDS] gaps

Provenance: When content moves: ## Slide 1: Title (source: slide 5) — author sees exactly what changed and from where.

9 / 14 Workflows

Multi-Persona Review & Synthesis

Parallel fan-out to N personas, merged into consensus findings and conflicts.

📄 Document
→
🔀 Fan-Out
→
👤👤👤 N Persona Reviews
→
🔄 Synthesis Pass
→
📋 Merged Report

Synthesis produces:

  • Consensus findings — issues flagged by multiple personas
  • Conflicts — weighted by each persona's known blind spots
  • Top-5 priority fix list — actionable next steps

Smart behaviors:

  • Single persona selected → synthesis pass skipped automatically
  • Review-informed rewrite → prior findings folded into rewrite prompt
  • "Advise" mode → draft proposals for each [NEEDS] gap

11 / 14 Assessment

Architectural Strengths

🎯 Single Source of Truth

One persona file → four consumers. Update once, propagate everywhere. No sync problems.

🔒 Local-First / No Lock-In

Ollama backend, Edge TTS. Runs entirely on-premises. No document content crosses external boundaries.

🔀 Composable Stages

Review → Rewrite → Narrate → Produce are independent. Stop at any point. No forced traversal.

🛡️ Graceful Degradation

No Review Lens? Infer. No persona? Ask. Missing data? [NEEDS:...]. Single reviewer? Skip synthesis.

📐 Consistent Schema

All 18 persona files are drop-in replaceable. No code changes. Web UI auto-discovers new files.

⚖️ Prompt Over Framework

Well-structured prompts carry load other systems push to LangChain/LangGraph code. Fewer dependencies, easier audit.

12 / 14 Assessment

Architectural Concerns

⚠️ Scorecard Incomparability

Scores not comparable across personas. A 4/5 on "Verdict placement" ≠ a 4/5 on "Trust Boundary Separation." Synthesis must operate on findings, not scores.

⚠️ Severity Definition Drift

"Critical" means different things per persona. Legal = regulatory exposure. Design = illegibility. Security = data leak. Users must understand this.

⚡ Context Window Pressure

Persona (~4K) + prompt (~1.5K) + document (variable). Large decks may hit attention limits on later slides.

⚡ Schema Coupling

Prompts assume persona file schema. No validation at upload. Relies on author discipline (currently holding across 18 files).

📝 No Version Tracking

Persona files are plain markdown. No mechanism to record which version produced which output.

📝 Synthesis Complexity

The merge of N heterogeneous reviews is the most complex reasoning step — with the least structural scaffolding.

13 / 14 Recommendations

Recommendations

Near-Term (Low Effort, High Value)

  • Add a "structure-preserving" flag to REWRITE — voice transformation without structural changes
  • Document the synthesis prompt — highest-risk reasoning step deserves visibility
  • Add persona file version headers — enables output provenance tracking

Medium-Term (Moderate Effort)

  • Build a persona compatibility matrix — document which pairs produce the best multi-lens reviews
  • Add "review the rewrite" auto-pass — verify rewrite meets 5/5 scorecard standard
  • Create lightweight schema validator — check required sections at upload time

Long-Term (Strategic)

  • Parameterize persona intensity — dial severity between 1 (gentle) and 5 (uncompromising)
  • Build review history layer — track trends across documents and teams
  • Consider domain-specialized persona forks — deeper expertise without expanding core library
14 / 14 Summary

Summary Verdict

A thoughtfully designed prompt orchestration system that punches above its architectural weight class.

✅ Strongest Element

The persona library — consistently structured, genuinely differentiated, self-aware about limitations, composable across four pipeline stages from a single source file.

✅ Key Trade-Off

Simpler orchestration (Express + subprocess) vs. framework-heavy alternatives. Fewer dependencies, more portable, easier to audit for government compliance.

The fundamental insight is correct: The persona file is the interface contract. The prompts define the task. The pipeline is just plumbing. That separation of concerns — stable interface, variable implementation — is what makes the system work and what will make it scale.
